あなたは既存の個人学習用Webサイト「medical-library」の改修を担当するエンジニアです。

今回の目的は、学習カード・リンク集サイトに「画像を紐づけて表示できる機能」を追加することです。

## 背景

このサイトは、医学・病理学習用の個人ライブラリとして使っています。
現在は主にテキスト、カテゴリ、リンク、学習進捗などを扱っています。

今後、ChatGPTなどで作成した図解・模式図・まとめ画像を、各学習項目に紐づけて表示したいです。

主な画像用途は以下です。

- 病理・医学の図解
- 自作またはAI生成の模式図
- 学習用まとめ画像
- 個人学習用の補助画像

患者個人情報を含む画像や、識別可能な症例画像を扱う想定ではありません。

## こちらの希望する大まかな方向性

画像本体は Firebase Storage に保存し、Realtime Database には画像そのものではなく、画像URLや保存パスなどのメタデータだけを保存する方針がよさそうだと考えています。

ただし、現在のコード構造との整合性を最優先してください。
既存設計を確認したうえで、より自然な設計があれば調整してください。

こちらの案はあくまで提案であり、最終設計は既存構造を見て判断してください。

## 現時点での想定構成

GitHub Pages:
- 画面表示を担当

Firebase Realtime Database:
- 既存の学習項目データを保存
- 画像URL、画像キャプション、Storage pathなどのメタデータを保存

Firebase Storage:
- 画像ファイル本体を保存

Google Drive:
- 今回の初期実装には含めなくてよい
- 必要になった場合のみ、将来的な原本保管先として検討する
- 初期設計に無理に組み込まない

## 実装したい最小機能

1. 学習項目に画像を1枚紐づけられる
2. 画像がある項目ではカード上にサムネイル表示される
3. 項目追加・編集画面で画像を登録・変更できる
4. ローカル画像をFirebase Storageへアップロードできる
5. アップロード後、取得した画像URLをRealtime Database側のitemに保存する
6. 画像キャプションを任意で保存できる
7. 既存のカテゴリ表示、検索、編集、学習進捗を壊さない

## データ設計の案

既存の item に以下のようなフィールドを追加する案を検討してください。

imageUrl:
- 表示用画像URL

imagePath:
- Firebase Storage内の保存パス
- 将来的な差し替えや削除に備えるため

imageCaption:
- 画像の短い説明
- 任意

ただし、既存のデータ構造により自然な名前や持たせ方があれば、それに合わせてください。

## Storageパスの案

例：

item-images/{itemId}/main.webp

または

item-images/{itemId}/{timestamp}.jpg

ただし、既存のitem IDやslug設計との相性を見て決めてください。

## UIの案

項目追加・編集画面に以下を追加する案を検討してください。

- 画像ファイル選択
- 現在の画像プレビュー
- 画像キャプション入力
- 画像削除または差し替え

初期実装では、1項目1画像で十分です。
複数画像対応は将来拡張で構いません。

## 表示側の案

item.imageUrl が存在する場合のみ、カード内に画像を表示してください。

希望する挙動：

- 画像がない項目は従来通り表示
- 画像がある項目だけサムネイルを表示
- loading="lazy" を使う
- スマホでも崩れない
- 画像クリックで拡大表示、または新しいタブで開けるとよい

## 画像サイズについて

初期実装では、アップロード画像をそのままFirebase Storageに保存する形でも構いません。

ただし、将来的には以下を検討したいです。

- WebP化
- 長辺1200〜1600px程度へのリサイズ
- サムネイル生成

初期段階で複雑になるなら、画像圧縮・変換は後回しにしてください。

## Security Rulesの方針

個人利用サイトなので、不特定多数が画像をアップロードできないようにしてください。

希望方針：

- 読み取りは、現在のサイト公開方針に合わせる
- 書き込みは認証済み管理者のみ
- 既存のadmins/{uid}のような管理者判定があるなら、それに合わせる

Realtime Database側の既存ルールや認証設計と整合するようにしてください。

## 実装方針

- 既存機能を壊さないことを最優先
- 大規模リファクタリングは避ける
- 最小変更で画像機能を追加する
- 現在のコード構造を確認してから、最適な実装方針を決める
- こちらの案より良い構造があれば提案・調整する
- Base64画像をRealtime Databaseへ直接保存する方式は避ける
